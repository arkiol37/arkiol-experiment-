## What Was Changed (v10 Schema Unification)

### 1. `apps/arkiol-core/prisma/_migrations_archive/` — **DELETED**

Before v10, Arkiol Core maintained its own Prisma migration history. During
the v9 consolidation those 12 migrations were moved to `_migrations_archive/`
and documented as "do not run." The archive has now been **permanently deleted**
because:

- Its presence implied the migrations could still be relevant
- The final consolidation migration (`20260306_v10_consolidation` in the shared
  folder) incorporated every table and column those migrations created
- Keeping deleted data in the repo creates confusion about what is canonical

### 2. `apps/arkiol-core/prisma/schema.prisma` — **Converted to fully inert stub**

The previous stub contained a `datasource db { ... }` block. While it had no
`generator` block (preventing `prisma generate`), a datasource-only schema is
still enough for the Prisma CLI to execute `prisma db push` and
`prisma validate`. That was an unacceptable footgun.

The file now contains **zero valid Prisma syntax** — only comments. The Prisma
CLI will fail with a parse error on any attempt to use this file as an input.
The VS Code Prisma extension still finds the file and suppresses "no schema
found" warnings; it treats a comment-only file as an empty schema.

### 3. `packages/shared/package.json` — **Added `"prisma"` field**

```json
{
  "prisma": {
    "schema": "./prisma/schema.prisma"
  }
}
```

The `"prisma"` field in `package.json` tells the Prisma CLI which schema to
use when invoked from that package directory without an explicit `--schema`
flag. Without it, `npx prisma` run from inside `packages/shared/` would search
upward through ancestor directories. With it, any accidental bare invocation
from that directory resolves to the correct schema immediately.

### 4. `scripts/deploy-checklist.sh` — **Fixed duplicate migration call**

Lines 106–116 ran `prisma migrate deploy` twice in a row against the same
schema, with the second call labelled "arkiol-core migration applied" — a
stale copy-paste from the pre-consolidation era when two separate schemas
existed. The duplicate call has been removed. There is now exactly one
`prisma migrate deploy` call in that script.

### 5. `apps/arkiol-core/README.md` — **Fixed bare `prisma generate` reference**

Line 127 described the Vercel `buildCommand` as `prisma generate && next build`
(without `--schema`). It now reads:
```
prisma generate --schema=../../packages/shared/prisma/schema.prisma && next build
```

### 6. `PRISMA.md` — **Updated to reflect inert stub and deleted archive**

- "The proxy stub" section now correctly describes a fully inert comment-only
  file with no valid Prisma syntax
- The directory tree no longer shows `_migrations_archive/`
- The "Do NOT do this" section explains that `--schema=apps/arkiol-core/...`
  now fails with a parse error (not just produces wrong output)

### 7. `packages/shared/prisma/migrations/migration_lock.toml` — **Added**

Prisma requires a `migration_lock.toml` file in the migrations directory to
track the database provider. Without it, `prisma migrate deploy` auto-generates
one on first run (benign, but creates an uncommitted file). The file is now
committed explicitly:

```toml
# Please do not edit this file manually
provider = "postgresql"
```

## Verification: How to Confirm the Single Source of Truth at Any Time

Run these commands from the monorepo root. All should succeed; none should
find any schema outside `packages/shared/`.

```bash
# 1. Confirm there is exactly one schema.prisma with real Prisma syntax
echo "=== schema.prisma files ==="
find . -name "schema.prisma" | grep -v node_modules
# Expected: TWO files —
#   ./apps/arkiol-core/prisma/schema.prisma      (inert comment-only file)
#   ./packages/shared/prisma/schema.prisma       (authoritative schema, 980 lines)

# 2. Confirm the core stub has zero valid Prisma syntax
echo "=== Core stub — should produce zero matches ==="
grep -E "^(generator|datasource|model|enum)\s" \
  apps/arkiol-core/prisma/schema.prisma
# Expected: (no output) — zero lines matched

# 3. Confirm the shared schema has the generator and datasource
echo "=== Shared schema — generator and datasource ==="
grep -E "^(generator|datasource)\s" \
  packages/shared/prisma/schema.prisma
# Expected:
#   generator client {
#   datasource db {

# 4. Confirm there is no migrations/ directory in arkiol-core
echo "=== arkiol-core migrations — should be empty ==="
find apps/arkiol-core/prisma -type d
# Expected:
#   apps/arkiol-core/prisma         (the directory)
# No migrations/ or _migrations_archive/ subdirectory

# 5. Confirm the only migration history is in packages/shared
echo "=== All migration.sql files ==="
find . -name "migration.sql" | grep -v node_modules
# Expected: only paths under ./packages/shared/prisma/migrations/

# 6. Confirm every script passes --schema= to Prisma
echo "=== Bare prisma commands (should be zero) ==="
grep -rn "npx prisma\b\|\"prisma " \
  package.json apps/arkiol-core/package.json \
  packages/shared/package.json \
  scripts/ .github/workflows/ \
  apps/arkiol-core/.github/ \
  --include="*.json" --include="*.yml" --include="*.sh" \
  | grep -v "\-\-schema=" | grep -v "^Binary\|node_modules"
# Expected: zero matches (every prisma command has --schema=)

# 7. Run prisma validate against the authoritative schema
npx prisma validate --schema=packages/shared/prisma/schema.prisma
# Expected: "The schema at ... is valid 🚀"

# 8. Confirm the migration_lock.toml is committed
cat packages/shared/prisma/migrations/migration_lock.toml
# Expected: provider = "postgresql"
```

## What Cannot Happen (By Design)

| Scenario | Why it cannot happen |
|----------|---------------------|
| Developer runs bare `prisma generate` from repo root | `package.json` `"prisma"` field is absent from root; Prisma will search upward but the first `prisma` field it finds (`packages/shared/package.json`) points to the shared schema |
| Developer runs bare `prisma migrate` from `apps/arkiol-core/` | `package.json` `"prisma"` field points to `../../packages/shared/prisma/schema.prisma` |
| Developer runs `prisma generate --schema=apps/arkiol-core/prisma/schema.prisma` | Fails with parse error — the file has no valid Prisma syntax |
| Developer runs `prisma db push --schema=apps/arkiol-core/prisma/schema.prisma` | Fails with parse error — no `datasource` block in that file |
| CI accidentally uses the core stub schema | Impossible — every CI step passes `--schema=${{ env.PRISMA_SCHEMA }}` where `PRISMA_SCHEMA: packages/shared/prisma/schema.prisma` |
| A new migration is created in the wrong directory | The only way to create a migration is `prisma migrate dev --schema=...`; the shared schema is the only one with a `generator` + `datasource` block |
| Schema drift between apps | Impossible — there is one schema, one client generation, one migration history. Both apps import from `@prisma/client` which was generated from the shared schema |
